# 数据集模块指南

## 目录结构

本模块用于组织来自不同来源的 CTF 挑战数据集。目录结构如下：

```
datasets/
├── __init__.py
├── preprocessor/           # 预处理工具
├── CLAUDE.md              # 本文件 - 每次必须阅读
├── <数据集名称>/           # 数据集来源（如 demo-xbow、xbow）
│   ├── droplet.yaml       # 数据集配置
│   ├── README.md          # 文档说明
│   └── challenges/        # 挑战目录
│       ├── CHALLENGE-001/
│       ├── CHALLENGE-002/
│       └── ...
└── ...
```

## 关键约束：挑战文件不可变

**⚠️ 绝对规则：挑战文件为只读，严禁修改。**

此规则适用于：
- 任何 `<数据集>/challenges/<挑战ID>/` 目录下的所有文件
- 任何操作者：人工、AI 代理、脚本或自动化流程
- 任何原因：修复 bug、改进、格式调整或"修正"

### 为什么要有这条规则

1. **数据完整性**：挑战是基准评估的基准数据
2. **可复现性**：修改可能导致现有评估或对比结果失效
3. **公平性**：所有评估必须在相同的挑战条件下进行
4. **可审计性**：原始挑战数据必须保留以供验证

### 实际操作指南

- **可以**：读取挑战文件进行分析或理解
- **可以**：将挑战文件复制到其他位置进行处理
- **不可以**：编辑、修改或"修复"任何挑战文件
- **不可以**：添加注释、格式变更或"改进"
- **不可以**：删除或重命名挑战文件
- **不可以**：运行会就地修改挑战文件的脚本

### 执行要求

- 此约束是绝对的，没有例外
- 如果挑战看起来有问题，报告问题而不是自行修复
- 对挑战文件的任何修改都视为严重错误
- 预处理器必须按原样处理挑战，不得修改源文件

## 添加新数据集

添加新数据集时：

1. 创建以数据源命名的目录（如 `my-source/`）
2. 添加 `droplet.yaml` 配置文件
3. 创建 `challenges/` 子目录
4. 将挑战目录放入 `challenges/` 中
5. 添加 `README.md` 记录数据源信息
6. **确保提交前所有挑战文件均为最终版本**

## 数据集配置（droplet.yaml）

每个数据集必须包含 `droplet.yaml` 文件：

```yaml
schema_version: 1
suite:
  id: <数据集ID>              # 唯一标识符
  name: <显示名称>            # 人类可读名称
  version: <版本号>           # 版本字符串
  license: <许可证类型>       # 许可证信息
auto_discover:
  - type: <适配器类型>        # 数据集加载器类型（如 xbow）
    path: challenges          # 挑战目录路径
    category: <分类>          # 挑战分类（如 web）
    task_type: <任务类型>     # 任务类型（如 web_ctf_online）
    default_max_submissions: <数量>  # 每个挑战的最大提交次数
```

## 预处理器用法

预处理器可将原始挑战数据转换为标准格式：

```bash
PYTHONPATH=. python -m datasets.preprocessor \
  --raw-path /path/to/raw/challenge \
  --output-dir datasets/drafts/my-suite \
  --challenge-id RAW-001
```

**注意**：预处理器从原始数据创建新的挑战文件，不会修改 `challenges/` 目录中已有的挑战文件。

## 当前数据集

- **demo-xbow**：包含 5 个 XBOW 挑战的演示数据集
- **xbow**：完整 XBOW 验证基准测试集（104 个挑战）

## 有疑问？

如果不确定某个操作是否违反不可变约束，请假设它违反并寻求确认。挑战数据的完整性至关重要。
